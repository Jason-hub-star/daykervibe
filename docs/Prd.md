# Hackerton Portal PRD

Project: Hackerton
Version: 1.0.0
Date: 2026-03-09 (KST)
Context: 이 문서는 재사용 가능한 해커톤 운영 포털을 구현하고 제출하기 위한 제품 기준 문서다.

## 1. 프로젝트 개요
### 1.1 문제 / 목표
- Problem: 해커톤 운영 경험은 대회 소개, 팀 구성, 제출 준비, 평가 이해가 여러 화면과 상태에 흩어져 있어 참가자와 운영자 모두에게 비효율적이다.
- Goal: `모집 -> 팀 구성 -> 제출 -> 평가 이해`를 한 제품 안에서 재사용 가능한 포털 형태로 제공한다.
- Non-goals:
  - 실시간 채팅 SaaS
  - 외부 DB 기반 운영 시스템
  - 관리자 백오피스 풀스택 구현

### 1.2 제품 한 줄 정의
- 표면적으로는 주어진 명세를 구현하는 대회 제출물이지만, 실제로는 바이브 코딩으로 해커톤 운영 제품을 어디까지 만들 수 있는지 보여주는 재사용 가능한 운영 포털 프로토타입.

### 1.3 타깃 사용자
- 참가자: 해커톤을 탐색하고 일정과 제출 방식을 이해해야 하는 사용자
- 팀장: 팀을 만들고 팀원을 관리하며 제출 준비를 이끄는 사용자
- 팀원: 모집글을 보고 합류하고 작전실에서 준비 상황을 확인하는 사용자
- 심사자: 공개 URL만으로 서비스 흐름과 완성도를 검토하는 사용자
- 운영자 관점: 다음 해커톤에도 재사용 가능한 구조를 보고 싶은 주최 측

### 1.4 Success Metrics
- 심사자가 로그인 없이 핵심 흐름을 3분 내 확인 가능
- 상세 페이지에서 필수 섹션 8개 모두 확인 가능
- 캠프/작전실/제출/랭킹 흐름이 끊기지 않음
- 문서/설명 점수 설득용 근거 문서가 준비됨

## 2. 핵심 사용자 경험
### User Story 1
- Given 사용자가 해커톤 정보를 모를 때
- When 메인 페이지에 진입하면
- Then 해커톤 보기, 팀 찾기, 랭킹 보기로 바로 이동할 수 있어야 한다

### User Story 2
- Given 사용자가 특정 해커톤에 관심이 있을 때
- When `/hackathons/:slug`에 들어가면
- Then 개요, 안내, 평가, 일정, 상금, 팀, 제출, 리더보드를 한 화면 맥락 안에서 이해할 수 있어야 한다

### User Story 3
- Given 팀장이 팀을 꾸리고 있을 때
- When `/camp`와 `/war-room/:teamId`를 사용하면
- Then 팀 모집, 팀 상태, 제출 준비 현황을 관리할 수 있어야 한다

### User Story 4
- Given 심사자가 결과를 검토할 때
- When 공개 URL에 접속하면
- Then 별도 키 없이 서비스 가치와 제출 준비 흐름을 바로 검수할 수 있어야 한다

## 3. 기술 스택 & 제약
### 3.1 고정 스택
- Framework: Next.js App Router
- Language: TypeScript
- Styling: Tailwind CSS
- Storage: `localStorage`
- Deploy: Vercel

### 3.2 환경 제약
- 외부 키 없는 검수 가능 상태 유지
- 더미 데이터와 로컬 상태 유지 전략 사용
- 모바일 기본 대응
- 저작권과 라이선스 준수
